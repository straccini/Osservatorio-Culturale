## SEZIONE 7 — Proposta strategica gestione fonti

### 7.1 Unificazione schema fonti

**Disallineamento attuale**:
- `FU_HEADERS` (Fonti_v1.js): 14 colonne
- `COL_F` (Bandi_v5.js, FontiBandi_v5): 18 colonne (+ `UltimoErrore, EnteDefault, UrlEnte, Livello, Note`, − `Tag`)
- `FontiPodcast` legacy: 10 colonne

**Fix runtime attuale**: `Fonti_v1.js:336` usa `findCol_` per matching header per nome (case-insensitive con sinonimi). Funziona ma è codice difensivo.

**Piano migrazione (3 step)**:

**Step 1 — Schema target 17 colonne**:
```
ID, Nome, URL, Tipo, Tag, Categoria, Priorita, Attiva,
DataAggiunta, UltimaScan, UltimoEsito,
NRecordTotali, NRecordUltimo, FailConsecutivi,
UltimoErrore, EnteDefault, Livello
```
(rimosso `UrlEnte`, `Note` → opzionali in JSON `extras`)

**Step 2 — Migrazione idempotente** in `FontiMigration_v1.js`:
- `migrateBandiV5ToFU17(dryRun)` — snapshot backup → riscrittura schema → mapping `Categoria` → `Tag` se vuoto
- `migratePodcastToFU17(dryRun)` — da 10col legacy a 17col

**Step 3 — Code cleanup**:
- Rimuovere fallback `findCol_`
- Eliminare `toggleFonteV5`, `getFontiBandiV5Admin` (già pianificato 🟢)

### 7.2 Agenti AI di scouting fonti

Nuovo file `Scouting_v1.js` con 3 endpoint:
- `scoutNewSourceFromUrl(url, tipoHint)` — Claude classifica URL, suggerisce metadata, pre-compila fonte
- `scoutSimilarSources(seedId, n=5)` — embedding similarity sulle fonti esistenti
- `scoutTrendingTopics(window='7d', tipo='bandi')` — cluster tematici da titoli recenti

UI: tab "Scouting AI" nel pannello admin. Flag `OC_SCOUT_ENABLED` in ScriptProperties.

## SEZIONE 8 — Proposta strategica digest intelligenti a 2 coorti

### Situazione attuale
- `sendDigestAuto()` (Codice.js:2381) **non distingue coorti**: stesso HTML per tutti
- 3 sorgenti utenti disponibili:
  - `MailingList` — lettori opt-in
  - `ContactsMatrix` — compilatori Matrix (con responseId → 10 dimensioni scoring)
  - `RichiestePrenotazione` — lead caldi (con `tematica_codice`, `stato_followup`)
- `Matrix_digest.js` ha già un sistema parallelo (`generateDigestForUser`, `generateDigestQueueAll`) — NON integrato con `sendDigestAuto()`

### Architettura proposta

```
sendDigestAuto()
  ├─ getDigestRecipientsByCohort()           NUOVO
  │    cohort A "generalista":  MailingList AND NOT in coorte B
  │    cohort B "lead caldo":   ContactsMatrix.optInSegmentato=true UNION
  │                              RichiestePrenotazione.stato_followup IN (caldo,contattato)
  │
  ├─ FOR each cohortA:  payload = buildGenericDigest(weekItems)
  │
  └─ FOR each cohortB:
        IF has responseId:    payload = generateDigestForUser(email, responseId)  [esistente]
        ELSE IF tematica:     payload = buildTematicDigest(items, tematica)        [NUOVO]
        ELSE:                 payload = buildGenericDigest                          [fallback]
        IF leadScore >= 30:   crm_notifyHotLeadDigested_ via Telegram
```

### Step concreti
1. Aggiungere `OptInDigestSegmentato` a `ContactsMatrix`
2. Aggiungere `OptInDigest` a `RichiestePrenotazione`
3. Nuovo modulo `Digest_routing.js` con `getDigestRecipientsByCohort()` + `buildTematicDigest()`
4. Refactor `sendDigestAuto` con loop per coorte
5. Estendere `DigestLog` con campo `Coorte` (generalista|tematico|matrixPersonalizzato)
6. Integrazione CRM_v1 scoring: `digest_sent +1pt`
7. Frontend admin: tab Generalisti | Tematici | Matrix-personalizzati nel pannello digest preview

### Cadenza proposta
- Generalisti: lunedì recap settimanale (come oggi)
- Lead caldi: lunedì + **mercoledì extra digest** focalizzato su news/bandi della settimana corrente sulla loro tematica
